# Landcover Class Normalization Fix

## Problem

Original issue: the `landcover_class` field in CSV data had floating‑point precision artifacts (e.g. `79.99999999999999`). The backend used `parseInt()` (truncation), which caused:
- `79.999...` → `parseInt()` → `79` (incorrect)
- `9.999...` → `parseInt()` → `9` (incorrect)

These invalid values (9, 19, 29, 39, 59, 79) caused:
- The frontend to display "Unknown" (because `LANDCOVER_NAMES` only defined 10/20/30/40/50/60/80/90)
- MaxMSP to receive invalid values, breaking timbre mapping

## Solution

Normalize `landcover_class` in the backend:

1. **Use `Math.round()` instead of `parseInt()`**: correctly handle floats
2. **Map to the nearest valid class**: ensure values stay within the valid range
3. **Define the valid class list**: `[10, 20, 30, 40, 50, 60, 70, 80, 90, 95, 100]` (includes `70` Snow/Ice)

## Code Changes

### server/index.js

Added the `normalizeLandcoverClass()` function:

```javascript
const VALID_LANDCOVER_CLASSES = [10, 20, 30, 40, 50, 60, 70, 80, 90, 95, 100];

function normalizeLandcoverClass(value) {
    // Preserve null/undefined/empty values - do NOT default to 10
    if (value == null || value === '' || (typeof value === 'string' && value.trim() === '')) {
        return null;
    }
    
    // Parse as float first, then round to nearest integer
    const num = parseFloat(value);
    if (isNaN(num)) {
        return null; // Invalid numeric value, preserve as null
    }
    
    // Round to nearest integer (handles 79.999... -> 80)
    const rounded = Math.round(num);
    
    // Treat values < 10 as missing data (ESA WorldCover classes start at 10)
    if (rounded < 10) {
        return null;
    }
    
    // If it's already a valid class, return it
    if (VALID_LANDCOVER_CLASSES.includes(rounded)) {
        return rounded;
    }
    
    // Find nearest valid class
    let nearest = VALID_LANDCOVER_CLASSES[0];
    let minDiff = Math.abs(rounded - nearest);
    
    for (const cls of VALID_LANDCOVER_CLASSES) {
        const diff = Math.abs(rounded - cls);
        if (diff < minDiff) {
            minDiff = diff;
            nearest = cls;
        }
    }
    
    return nearest;
}
```

Used in `parseCSV()`:

```javascript
} else if (h === 'landcover_class') {
    // Normalize to valid ESA WorldCover class (handles float precision issues)
    row[h] = normalizeLandcoverClass(val);
}
```

Also used in `calculateViewportStats()` (defense in depth) to normalize values when needed, preserving null for missing data.

## Result

### Before
- `79.99999999999999` → `parseInt()` → `79` (incorrect)
- `9.99999999999993` → `parseInt()` → `9` (incorrect)
- Frontend displayed "Unknown"
- MaxMSP received invalid values

### After
- `79.99999999999999` → `Math.round()` → `80` → valid
- `9.99999999999993` → `Math.round()` → `10` → valid
- Frontend displays the correct class name
- MaxMSP receives valid values

## Important Notes

**Empty/null values are preserved as null** - they are NOT defaulted to 10 (Tree/Forest). This ensures:
- Data accuracy: missing landcover data is correctly represented as missing
- Percentage calculations: only grids with actual landcover data are counted in percentages
- No data skewing: defaulting to 10 would incorrectly inflate Tree/Forest percentages

**Behavior**:
- Float precision artifacts (79.999... → 80) are correctly rounded
- Invalid values are mapped to nearest valid class (9 → 10, 19 → 20, 79 → 80)
- Empty/null/undefined values return null (preserved as missing data)
- Values < 10 are treated as missing data (return null)
- Valid values remain unchanged
