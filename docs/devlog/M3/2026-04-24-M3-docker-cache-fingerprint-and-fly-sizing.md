# 2026-04-24 — Fix: Docker Cache Fingerprint + Fly VM Sizing

Two production-only problems surfaced on the first `fly deploy`:

1. **Pre-warmed spatial-index cache was rejected at runtime** — the
   fingerprint check included `fs.statSync().mtimeMs`, which Docker's
   layer extraction does not preserve, so every cold start invalidated
   the cache and re-parsed 21 MB of CSVs (~60-90s on Fly's shared CPU).
2. **256 MB VM OOM-killed node during that re-parse** — Alpine + node
   20 allocates ~828 MB of virtual address space at startup; even with
   139 MB resident, Fly's 256 MB cgroup limit tripped the kernel OOM
   killer before the spatial index finished loading.

Local dev never hit either issue: the cache was generated by the same
filesystem it runs against, and there was no 256 MB cap.

## Fix 1: drop mtime from the CSV fingerprint

`calcCsvFingerprint` in [`server/normalize.js`](../../../server/normalize.js)
now hashes `filename + size + head-and-tail content snippet` only.
Content snippet catches every real edit (re-export, manual patch,
different file with same size); size catches trivial cases before we
read. mtime was noise — it drifted with image extraction, `cp -p`,
rsync, git checkouts across machines, etc., forcing needless CSV
re-parses that had no functional payoff.

After the fix, the second Fly deploy showed:

```
[Cache] Loading from all_grids.json (67331 grids)
[Normalize] Loaded params (fingerprint+aggregation matched) v=v2_area_weighted
[Index] Built spatial index: 67331 buckets (GRID_SIZE=0.5°)
Server listening on port 8080 (HTTP + WebSocket)
```

Cold-start time dropped from "OOM before completion" to ~4 seconds
end-to-end (VM boot + cache load + index build).

## Fix 2: bump Fly VM memory 256 MB → 512 MB

The spatial index itself peaks around 150-200 MB resident, comfortably
inside 512 MB but over the 256 MB cgroup limit once Alpine's virtual
address accounting is included. `fly.toml` raises `memory = '512mb'`
and relaxes the `/health` check's `grace_period` from 30s → 60s so the
first request after a fresh machine start has slack while the cache
loads from disk.

Cost impact: the 512 MB shared-1 VM runs ~$3.89/month, still inside
Fly's $5/month free credit for solo developers (roughly ~80% of the
budget at 24/7 uptime; `auto_stop_machines = "suspend"` keeps the
actual duty cycle well below that).

  the hash input; docstring updated.
