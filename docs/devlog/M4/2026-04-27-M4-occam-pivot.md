# 2026-04-27 — Design: Occam Pivot — Drop P0-5, P1-2, P2-1/2/3; Collapse P3/P4 Stages; Remove Unused `lerp`

After P0-1 / P0-2 / P0-3 / P0-4 / P0-5 landed on `feat/M4`, an internal Occam-razor audit identified that the original M4 plan was itself violating its stated principle in several places — adding new abstractions and files for hypothetical future need, rather than addressing the actual stated problem (audio-engine.js too large; server/index.js mixing responsibilities). The plan was re-cut from 29 stages to 22 by rolling back P0-5 and dropping six other not-yet-started stages plus several intra-stage redundancies.

## Why now

Halic's question, paraphrased: "本来说是奥卡姆剃刀的，怎么感觉冗余组件越来越多了？怎么绕了一圈，代码库更加庞大了？" Honest answer: P0-5 as shipped (recorder HTML 280 lines + baseline-notes.md 100 lines + 7 proposal edits + a 200-line devlog) delivered ~600 lines of added code/docs for ~zero downstream value — the recorder is unused, the benchmark numbers can be regenerated by `npm run benchmark` in 30s, and the proposal softening was needed only because the original plan over-specified.

A wider audit of the unstarted stages found the same pattern in P1-2, P2-1, P2-2, P2-3, and the 9-file P3 split. By the proposal's own LOC accounting, M4 was set up to add ~20 net new files while delivering ~60-line LOC drop — a ratio that betrayed the "razor" framing.

## Decisions

### Rolled back

- **P0-5 (entire stage)**: deleted `scripts/record-audio-baseline.html`, `docs/plans/M4/baseline/baseline-notes.md`, the P0-5 devlog `2026-04-27-M4-baseline-tool-only.md`. No audio reference will be captured during M4. Audio regression detection relies on §2.E manual A/B listen + unit-test routing topology coverage + DevTools-eval numeric verification (the kind P0-3 already demonstrated with `dbToLinear(12) === Math.pow(10, 12 / 20)`). P5-1 captures its own at-time idle CPU baseline rather than referencing a P0-5 number.

- **P0-3 `lerp`**: removed `lerp` export from `frontend/audio/utils.js` + the corresponding `describe('lerp', …)` block from the test. It had zero callers; it was added because the original proposal listed it as P0-3 deliverable, but CLAUDE.md "no design for hypothetical future requirements" applies. P3-3 (raf-loop EMA driver) can add `lerp` directly to `audio/utils.js` at the moment it's first needed — the diff is one function.

### Dropped from forward plan

- **P1-2 (`server/http-client-state.js` extraction)**: existed only to feed P4-3 (originally P4-4); the merger should happen directly. The "shared helper module" was a stepping stone for an architecture decision better made at the merger point.

- **P2-1 (`frontend/utils.js` generic helpers)**: `clamp` / `debounce` / `querySelectorRequired` were going to be extracted into a shared module — but the audit didn't first count actual call sites for these patterns in non-audio frontend code. Defer until ≥ 3 concrete call sites exist.

- **P2-2 (`frontend/event-bus.js`)**: a 40-line pub/sub plus a 7-row event roster decision. The current direct-callback wiring works; the bus would help in P2-5 / P3-3, but those stages are themselves dropped or simplified. Skip until a concrete second consumer materializes.

- **P2-3 (`frontend/lifecycle.js`)**: a three-state state machine (`bootstrap → ready → interactive`) with illegal-transition errors, replacing what is currently 5 lines of sequential init in `main.js`. Exactly the over-engineering CLAUDE.md warns against. main.js can stay sequential.

- **`_statsCounter` SIGTERM rewrite + `config.js` IIFE simplification** (originally part of P1-3): cosmetic; defer to M5 if ever.

### Collapsed

- **P3 (audio decomposition): 7 stages / 9 final files → 5 stages / 4 new files** (`context.js`, `buffer-cache.js`, `raf-loop.js`, `engine.js`). Swap-timer / BusVoice / LoopSlot / announcer routing stay inside `engine.js` as named exports — testable via the P0-1 audio-context mock without needing separate module boundaries. The pure-function boundary (raf-loop) is the testability win that genuinely justifies a separate file; the rest are fine inside `engine.js`. Targets `engine.js` ≤ 800 lines (down from 1186, with most of the bulk preserved as named exports for testability).

- **P4 (server decomposition): 4 stages → 3 stages.** Drop the `bootstrap.js` extraction (`server/index.js`'s startup block is naturally cohesive). Keep routes.js, ws-handler.js, and the mode/delta merger.

- **`server/index.js` target: ≤ 100 → ≤ 250 lines.** The "≤ 100" threshold was cargo-cult; 250 is a more honest number for "extract the routes and the WS handler, leave the express setup + bootstrap inline." A 200-line entry point is fine.

- **Per-stage devlog → per-significant-stage devlog.** "Significant" means: new module file, deletion > 50 lines, architectural change, or anything that benefits from a "what nodes connect to what" review. Trivial code moves (popup → popup.js, progress → progress.js, types.js delete) share a phase-close devlog. M4 ledger goes from 30 entries to ~12-15.

### Kept (still genuine wins)

- P1-1 spatial bucket-loop collapse + dead `gridData.filter` removal (perf + dead code).
- P1-3 lcFractions memoization (P1-4 in original numbering; renumbered after P1-2 dropped) — measurable perf win; baseline numbers are in the prior P0-5 devlog before that file was deleted, but `npm run benchmark` regenerates them.
- `server/types.js` deletion (P1-2 in new numbering, was P1-3a) — 104 dead lines.
- P3-0 lazy-init audio-engine — the AudioContext-instantiation hazard during decomposition is real; this stage stays.
- All P5 stages.

## Net effect on M4 scope

| Metric | Pre-pivot | Post-pivot | Delta |
|---|---:|---:|---:|
| Total stages | 29 | 22 | -7 |
| Remaining hours | 64-73h | ~47-49h | -27% |
| New files added (estimated) | ~20 | ~9 | -55% |
| Final `audio/` file count | 9 | 6 | -3 |
| `audio-engine.js` final state | deleted | deleted | unchanged |
| Devlog entries | 30 | ~12-15 | -50% |

`audio-engine.js` still gets deleted at P5-4 — the headline goal is unchanged. `server/index.js` still drops dramatically. The cuts are about removing the **abstraction surface** that the proposal would have ADDED on top of those wins.
