### One deferred CPU measurement

The P5-1 §11 row asked for a **before/after DevTools Performance recording** to verify ≥ 30% main-thread CPU drop on idle audio. But "structural argument" ≠ "measured drop", so this target is **deferred to M5**, where a deterministic reproduction of the buffer-load race (synthetic timing harness) can land alongside a CPU measurement.

## 6. The P5-1 revert + redo

The single non-trivial production incident in M4. Worth recording as the canonical example of why the proposal §2.E "manual A/B listen" gate is non-negotiable.

**Original P5-1** (`224b1ca`): added `isEmaIdle()` + idle suspension to the rAF loop, with five wake triggers covering every external EMA target mutation. Vitest passed. Preview-environment dry-run passed. **Production:** seven-bus ambience went silent on hard-reload Start audio.

**Diagnosis** (recorded in [`2026-04-27-M4-revert-p5-1-raf-idle.md`](../../devlog/M4/2026-04-27-M4-revert-p5-1-raf-idle.md) and [`2026-04-27-M4-raf-idle-detection-redo.md`](../../devlog/M4/2026-04-27-M4-raf-idle-detection-redo.md)): per-bus `gain.value` writes are gated on `bufferCache.has(i)`. On a cold buffer load (~3–5 s), EMAs converged within `IDLE_THRESHOLD = 0.001` before any sample finished decoding, the rAF self-suspended, and the gate was never re-evaluated. `gain.value` stayed frozen at the initial 0 forever. The five-wake enumeration was complete with respect to **target writes** but missed **buffer-load completion** as a separate readiness condition.

**Hotfix attempt** (`151c56c`, also reverted): tried removing the `bufferCache.has(i)` gate. Didn't help — sources weren't connected at write time, so the writes had no audible effect; subsequent ticks that *would* matter never happened because the rAF was suspended.

**Revert** (`bcb7ab2`, `35d6058`, then narrative devlog `65394d2`): rolled both commits back to the P4-3 endstate.

**Redo** (`6bc2a8d`): added the **sixth wake trigger** — `startRaf()` at the end of `start()` after `await bufferCache.loadAll(audioCtx)`. Plus a defensive `if (!audioCtx || suspended) return` guard at the top of `startRaf()` to fix a sibling latent bug (a `updateMotion()` called pre-`start()` would `requestAnimationFrame` against a null `audioCtx`, leaving a stale `rafId` that short-circuited the real `startRaf()` later). Vitest still synchronous, so the unit tests still couldn't reproduce the original failure — but the structural argument plus a preview-environment probe (gain values written to the converged EMA target after start, engine rAF absent from the `requestAnimationFrame` caller stack post-convergence) was enough to ship.

**Lessons recorded for M5:**

1. The vitest `audio-context-mock` is fully synchronous; `bufferCache.loadAll` resolves on the same microtask. The bug requires `loadAll` to take seconds — only reproducible in real-browser cold-load conditions. A deterministic CI fixture (synthetic `loadAll` delay + assert `gain.value` non-zero by `start()` resolution) is the right backstop and is the natural backbone for the deferred CPU measurement work in §5.
2. The `main.js` WS-onOpen vs Mapbox style.load race noted in the original revert devlog is unrelated to P5-1 but is in the same hazard class. Open M5 item.
3. Performance optimizations that introduce a state machine (idle ↔ active) need to enumerate **all readiness conditions** the loop is gating on, not just the obvious target-write conditions. The buffer-load gate was implicit in the per-bus `if (gains[i] && bufferCache.has(i))` line and easy to miss.
