## 1. Theme

M6 builds **hover-glow**: cells brighten softly when both the cursor *and* a country border or coastline are nearby. The feature is a wayfinding signal that complements M3's city-name announcer — the announcer says *what* place you're hovering on, the hover-glow shows *where the borders are* without forcing the user to read the basemap.

The milestone splits cleanly into three architectural eras separated by a mid-milestone pivot.

**Era A (2026-04-29) — CPU-side, feature-state-driven runtime.** A 67k-cell linear scan in `tick()` computes per-cell glow, writes `setFeatureState({glow})` on candidates, walks `prev \ new` and zeros leavers. Paint expression interpolates `circle-color` from `#606060` → `#FFFFFF` over the feature-state value. Three invariants (drag-lag fix, no square-shape leak, no progressive degradation) directly enumerate the failure modes of the rolled-back earlier attempt. Three same-day fix iterations tighten the visual (250 km → 40 km border falloff for single-row line not 5-row swath), eliminate idle CPU burn (RAF-coalesced tick), and recover sub-frame freshness during drag (synchronous tick, spatial-bucket index, 1.7× wider radius). End of era: the visible behavior is right but interaction "卡卡的".

**Pivot (2026-04-30 mid-day) — GPU custom-layer rewrite.** Diagnosed (with browser instrumentation) that JS-side per-frame work is sub-millisecond; the bottleneck is **GPU vertex-buffer upload churn** triggered by ~150–200 `setFeatureState` writes per drag-frame. The first attempt at this — pause `tick()` during drag, take a catchup tick on `moveend` — froze the glow geographically while dragging. Replaced wholesale by a single Mapbox custom WebGL layer: one VBO upload at init time (67k × 32 B = ~2.1 MB), per-frame only a `vec2 cursor` uniform updated, glow computed in the fragment shader. `frontend/hover-glow.js` shrinks 819 → 293 LOC after the rewrite (further to 177 LOC after the second Occam sweep). Spatial-bucket index, SoA candidate buffer, drag-pause flag, `setFeatureState` cleanup machinery — all deleted at this step; JS curve helpers stayed and were removed by the second Occam sweep.

**Era B (2026-04-30 afternoon) — projection fixes + Occam sweep.** Three projection bugs (ECEF unit-sphere vs Mapbox `GLOBE_RADIUS`, `mix()` argument reversal, mercator-mode `transition` default polarity) caught by the user looking at zoom-2 globe and reporting "halo floats off to the left". Halo zoom curve replaced (single scalar 3.0 → zoom-stops table because halo-to-cell-spacing ratio diverged at low zoom). Second Occam sweep removes everything the GPU rewrite made dead — JS curve helpers and their tests, `lerpStops`'s "circular import" non-comment, `_visible` flag (replaced with sentinel cursor), 5 other smaller items. Module reaches steady state.

The milestone has not closed (no merge to `main`); the rest of the verification list is in §5.

The milestone's central architectural decision. ## 4. Key design decisions

1. **The three explicit invariants from the rolled-back attempt** — drag-lag fix (re-`unproject` on every render with current transform), no square-shape leak (multiplicative radial scalars), no progressive degradation (diff/cleanup every frame). Setting these as named invariants made the GPU rewrite easier later: each invariant either *survived* the rewrite (#1 by uniform update; #2 by GLSL math being purely radial) or was *made obsolete* (#3 because there's no per-cell state to clean up — the fragment shader recomputes from scratch every frame).
2. **The `promoteId` rule (build-time only, never both)** — pinned at the data layer with two source-level tests. Any change in either direction has to update both files together.
3. **Additive cursor floor over `Math.max`** — preserves C¹-continuity of the falloff curve. The first sketch (`Math.max(bf, cursorFloor)`) introduced a kink at the crossover, contradicting invariant #2's spirit.
4. **Static calibration of border falloff** — 250 km → 40 km cap collapsed swath-to-line because `GRID_SIZE = 0.5°` cell spacing dictates that "border-passes-through" is a 0–28 km range. Cap should be just past the corner-to-centroid diagonal (~39 km). Same logic governs every tunable in `frontend/config.js`: defaults tuned to the data's geometry, not generic UI heuristics.
5. **GPU custom-layer rewrite as the right answer.** The CPU-side iteration (sync tick → spatial bucket → SoA → drag-pause) was attacking a JS bottleneck that profiling showed wasn't the real problem. The bottleneck was **GPU vertex-buffer upload churn** — invisible to JS-side timing because it lands on the next compositor frame, not the current JS turn. Rewriting as a custom WebGL layer is the idiomatic GPU path: VBO uploaded once, per-frame work is one `vec2` uniform update + a fragment-shader pass over 67k point sprites that the GPU is built for. **Decision criteria that justified the rewrite over more CPU optimization**: (a) the user's own framing — "彻底换实现思路的方法"; (b) every CPU optimization landed had been a bug-hunt, not a feature; (c) the rewrite would *delete* code, not add it (819 → 150 LOC); (d) a custom-layer scaffold already lived in M3's `applyMinimalBasemap()` precedent for "talk to Mapbox at the GL level."
6. **Two Occam sweeps, both bundled per Occam-Group policy.** First sweep (cleanup-sweep) cut three high-confidence items + measured to skip two micro-optimizations. Second sweep (occam-sweep) was post-rewrite — the GPU path made a class of code dead that wasn't dead before. The "JS curve helpers locked nothing because no test cross-checks JS vs GPU" insight is the canonical example of why the second sweep was justified, and it's also the source of the deferred follow-up (puppeteer screenshot test).
7. **Sentinel cursor `(999, 999)` over `_visible` flag.** Replacing a render-time conditional with an in-band signal that the math already handles correctly. The sentinel falls outside any realistic globe coordinate; `cursorFactor` returns 0 everywhere; the `discard` path culls the frame. Less branching, fewer code paths to verify.
8. **Halo-to-spacing ratio, not halo-to-dot ratio.** The user's "thick" complaint at low zoom was the data telling us the wrong invariant was being preserved. Single-scalar `HALO_SCALE` keeps halo-to-dot constant; what matters perceptually is halo-to-spacing. Zoom-stops table keeps the latter approximately constant (tighter at low zoom where dots crowd, larger at high zoom where dots are sparse).
9. **GLSL coupling enforced by throw, not silent truncation.** `MAX_FALLOFF_STOPS = 4` is coupled to `uniform vec2 uFalloff[4]` + `for(i=0;i<3;i++)` + the const itself. None are cross-checked at runtime. Throwing on over-cap names all three GLSL spots that need to grow together — a contributor who hits the throw is forced to find the call sites the silent path would have hidden.

## 6. What M6 sets up for downstream milestones

- **Custom WebGL layer template.** `HoverGlowLayer` is the project's first instance of `CustomLayerInterface`. The pattern (init-time VBO upload + per-frame uniform updates + globe/mercator `mix()` blend) is reusable for any future visual that needs per-cell state independent of Mapbox's vector-tile pipeline. Candidates: per-cell ambient-mix visualization, audio-bus heatmap, city-announcer pop-up trail.
- **Border-distance dataset.** `border_dist_km` is a per-cell scalar that any downstream feature can consume — sound design (proximity-to-coast as a signal), UI (highlight border-adjacent cells), even server-side queries. PMTiles + sidecar pipeline shows how to bake derived geometry into the data layer without breaking the existing wire format.
- **Sidecar binary format precedent.** `GSIDX001` magic + count + gridSize + interleaved typed-array body is a pattern that scales to any future per-cell typed-array payload. Frontend's dual-view technique (one buffer slice, two `TypedArray` views) is documented in the format spec.
- **The `promoteId` rule** is now a project-wide invariant pinned by source-level tests. Future tile schema changes have a precedent for how to settle build-time vs run-time `feature.id` ownership.
- **Live-tunable feature pattern.** `window.__hg.tune({...})` (DevTools edit → next render → see result, no reload) is the third instance of this pattern in the project (previous: `localStorage.ENABLE_LOUDNESS_NORM` from M3, M4 `audio-engine.js` exposes via the engine's own controls). Worth lifting into a shared convention if a fourth feature wants it.
- **Two-Occam-sweep cadence** — first sweep right after the feature lands ships, second sweep after the next significant architectural change. Pattern recorded in the user's memory ([feedback_occam_single_commit.md](../../../.claude/projects/-Users-halic-Installed-geo-sonification/memory/feedback_occam_single_commit.md)) and confirmed working here.
